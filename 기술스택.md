전부 오픈소스로 공개되어 있고, 핵심 서버 측 구현 언어는 거의 모두 Python입니다. 다만 Horizon은 웹 대시보드라서 Python(Django) 중심이면서 HTML/CSS/JavaScript도 함께 사용합니다. OpenStack 커뮤니티는 각 프로젝트의 주 저장소를 OpenDev에 두고, GitHub의 openstack/* 저장소는 그 미러(read-only mirror) 라고 안내합니다.

정리하면 아래처럼 보시면 됩니다.

역할: 인증, 권한, 서비스 카탈로그

공식 저장소: opendev.org/openstack/keystone

GitHub 미러: github.com/openstack/keystone

역할: VM 생성/스케줄링/컴퓨트 관리

공식 저장소: opendev.org/openstack/nova

GitHub 미러: github.com/openstack/nova

스케줄러 코드도 Python 파일 기준으로 공개되어 있습니다.

Neutron: 주로 Python

역할: 가상 네트워크, 서브넷, 라우터, Floating IP

GitHub 미러: github.com/openstack/neutron

현재 코드와 문서 모두 공개되어 있습니다.

Horizon: Python + Django 기반, 프런트엔드에 HTML/CSS/JavaScript 사용

공식 저장소: opendev.org/openstack/horizon

OpenStack의 코어 서비스는 거의 전부 Python 생태계 위에서 만들어졌고, Horizon만 웹 UI 성격 때문에 Django + 프런트엔드 기술이 함께 들어간다고 보시면 됩니다.

###

이제는 각 컴포넌트를 “무슨 프로세스가 떠 있고, 어디에 상태를 저장하고, 서로 어떻게 통신하는가” 기준으로 보면 이해가 훨씬 빨라집니다.

전체 그림

OpenStack 코어 서비스들은 대체로 이런 공통 구조를 가집니다.

REST API 서버

사용자나 Horizon, CLI가 호출

DB

상태 정보 저장

메시지 큐(RPC)

서비스끼리 비동기/내부 명령 전달

백엔드 에이전트/워커

실제 호스트나 네트워크 장비, 스토리지에 작업 수행

특히 Nova는 공식 문서에서 API 서버가 DB read/write를 수행하고, 필요 시 다른 Nova 서비스로 RPC 메시지를 보내는 “messaging-based shared-nothing architecture”라고 설명합니다. 설치 가이드도 OpenStack 전반에서 SQL DB와 message queue를 공통 기반으로 둡니다.

1) Keystone

역할: 인증, 토큰, 프로젝트(테넌트), 사용자, 역할(Role), 서비스 카탈로그

실행 구조

주로 Identity API 서버

다른 OpenStack 서비스가 요청이 들어오면 Keystone에 토큰 검증을 의존

DB

기본적으로 SQL 기반 설정/상태 저장

일부 항목은 LDAP 연동도 가능하다고 문서에 나옵니다.

메시지 큐

Nova/Neutron/Cinder처럼 “메시지 큐 기반 분산 워커” 성격은 상대적으로 약합니다.

핵심은 인증/권한 API 서비스라는 점입니다.

기술 포인트

다른 모든 서비스가 auth_strategy = keystone 식으로 Keystone과 연계

실무에서는 “OpenStack의 IAM/SSO 중심축”으로 보면 됩니다.

REST API 성격

토큰 발급

사용자/프로젝트/역할 관리

서비스 카탈로그 조회

즉, Keystone은 분산 작업 엔진이라기보다 OpenStack 전체의 신원·권한 관문입니다.

2) Glance

역할: VM 이미지 관리

실행 구조

공식 문서는 Glance를 client-server architecture라고 설명하고, 사용자 요청은 REST API로 들어옵니다.

DB

이미지 자체 바이너리와 별개로

이미지 메타데이터

상태

위치 정보
등을 중앙 DB에 저장합니다. Glance 문서는 기본적으로 SQL 기반 DB를 사용한다고 설명합니다.

실제 이미지 저장소

이미지 파일은 DB에 직접 넣는 게 아니라 보통

로컬 파일시스템

Ceph

Swift

기타 backend
쪽에 저장하며, 이 접근은 glance_store 계층이 담당합니다.

메시지 큐

Nova/Cinder처럼 RPC 중심 구조가 핵심은 아닙니다.

주된 역할은 이미지 API + 메타데이터 관리 + 백엔드 스토어 연동입니다.

REST API 성격

이미지 업로드

이미지 목록/상세 조회

이미지 다운로드

이미지 상태 변경

즉, **Glance는 “이미지 레지스트리 + 이미지 스토어 게이트웨이”**에 가깝습니다.

3) Nova

역할: VM 생성, 삭제, 스케줄링, 컴퓨트 자원 제어

실행 구조

Nova는 OpenStack에서 가장 전형적인 분산 백엔드 아키텍처입니다.

대표 프로세스 관점으로 보면:

nova-api

nova-scheduler

nova-conductor

nova-compute

공식 문서에 따르면 API 서버는 REST 요청을 처리하면서 DB read/write를 하고, 필요하면 다른 Nova 서비스에 RPC 메시지를 보냅니다. 또한 Nova는 message queue 기반 shared-nothing architecture를 사용합니다.

DB

인스턴스 상태

스케줄링 관련 상태

셀(cell) 구조 정보
등을 DB에 저장합니다. 최신 Cells v2 문서도 API DB, cell DB, cell0 DB 구조를 설명합니다.

메시지 큐

Nova 핵심입니다.

서비스 간 내부 통신에 RPC/message queue를 사용합니다. 공식 문서는 oslo.messaging 기반이라고 설명합니다.

실제 작업 수행

nova-compute가 하이퍼바이저(KVM 등)와 붙어 실제 VM 생성/삭제 수행

nova-scheduler가 어느 호스트에 VM을 올릴지 결정

REST API 성격

서버 생성/조회/삭제

flavor, keypair, metadata, console 등 관리

즉, **Nova는 “OpenStack의 VM 오케스트레이터”**이고, 내부적으로는 API + DB + MQ + compute worker 구조를 가장 잘 보여주는 컴포넌트입니다.

4) Neutron

역할: 가상 네트워크, 서브넷, 라우터, 포트, Floating IP

실행 구조

Neutron의 핵심 프로세스는 **neutron-server**입니다. 보안 가이드와 설치 문서 모두 neutron-server가 Networking API를 노출하는 Python daemon이라고 설명합니다.

또한 Neutron은 실제 데이터플레인 처리를 위해 여러 agent와 연동합니다. 예를 들면:

L2 agent

L3 agent

DHCP agent

metadata agent

OVS/OVN 관련 컴포넌트

DB

네트워크 상태와 플러그인 상태를 저장하는 DB를 사용합니다. 문서에서도 neutron-server와 각 agent 사이 라우팅 정보 및 네트워크 상태 저장을 위해 DB를 쓴다고 설명합니다.

메시지 큐

공식 개요 문서에서 neutron-server와 various agents 사이 정보를 전달하는 데 message queue가 널리 사용된다고 설명합니다.

실제 작업 수행

neutron-server는 “명령/정책/상태 관리”

각 네트워크 노드/컴퓨트 노드의 agent가

브리지 생성

라우팅 반영

NAT/Floating IP 처리

DHCP 반영
같은 실제 네트워크 작업 수행

REST API 성격

network / subnet / router / port / floating IP / security group 관리

즉, **Neutron은 “네트워크 컨트롤 플레인”**이고, 실제 스위칭/라우팅 반영은 agent 또는 OVN/OVS 계층이 담당합니다.

5) Cinder

역할: 블록 스토리지(볼륨) 생성, 연결, 스냅샷, 타입 관리

실행 구조

공식 문서는 Cinder 구성요소로

cinder-api

cinder-scheduler

cinder-volume
를 설명합니다.
cinder-api는 API 요청을 받고, cinder-scheduler는 적절한 백엔드를 선택하고, cinder-volume은 실제 스토리지 백엔드 드라이버와 상호작용합니다.

DB

볼륨 상태

스냅샷

타입

백엔드 관련 메타데이터
등을 DB에 저장합니다. 설치 문서도 Cinder DB 설정을 별도로 요구합니다.

메시지 큐

Cinder는 메시지 큐를 명시적으로 사용합니다.

공식 문서에 cinder-volume이 다른 프로세스와 message queue를 통해 상호작용한다고 나옵니다. HA 문서도 API→Scheduler, Scheduler→Volume 식의 MQ 사용 패턴을 설명합니다.

실제 작업 수행

스토리지 장비나 소프트웨어 백엔드와 드라이버 방식으로 연결

예: LVM, Ceph, SAN/NAS 벤더 스토리지 등

REST API 성격

volume 생성/삭제

attach/detach

snapshot

type / quota 관리

즉, **Cinder는 “스토리지 컨트롤 플레인 + 드라이버 프레임워크”**입니다.

6) Horizon

역할: 웹 콘솔

실행 구조

Horizon은 다른 컴포넌트처럼 MQ 중심의 분산 엔진이 아니라, 공식 설명대로 Django-based project입니다. 즉 웹 애플리케이션입니다.

DB

일반적인 의미의 대규모 서비스 상태 DB를 Horizon이 직접 주도하는 구조는 아닙니다.

실제 클라우드 상태는 Keystone/Nova/Neutron/Cinder 등의 API 뒤편 DB에 있습니다.

Horizon은 그 API들을 호출해 화면에 보여주는 역할입니다.

메시지 큐

핵심 구조에 MQ가 중심은 아닙니다.

실제 동작

사용자가 웹에서 클릭

Horizon이 각 OpenStack 서비스의 REST API 호출

결과를 Django 화면으로 렌더링

REST API 성격

Horizon 자체가 “클라우드 리소스 원천 API”라기보다

각 서비스 API의 웹 프런트엔드입니다.

즉, Horizon은 OpenStack의 콘솔 UI 계층입니다. AWS로 치면 AWS Console에 가장 가까운 위치입니다.

한 번에 연결해서 보면

사용자가 Horizon에서 **“VM 하나 생성”**을 누르면 대략 이런 흐름입니다.

Horizon이 Nova API 호출

Nova가 Keystone 토큰 확인

Nova가 DB 기록

Nova Scheduler가 어느 컴퓨트 노드에 올릴지 결정

Nova가 MQ로 nova-compute에 작업 전달

필요 시 Neutron에 네트워크 포트 요청

필요 시 Cinder에 볼륨 연결 요청

부팅 이미지가 Glance에서 제공됨

최종 상태가 각 서비스 DB에 반영되고 Horizon에 표시됨

이 구조 때문에 OpenStack는 단순한 앱이 아니라 여러 API 서버 + DB + MQ + agent가 협업하는 내부 클라우드 운영체제처럼 보시면 됩니다.

AWS 개념으로 아주 짧게 대응시키면

Keystone = IAM + 인증 게이트

Glance = AMI 저장소 비슷한 개념

Nova = EC2 제어면

Neutron = VPC / Subnet / Router / SG / Public IP 제어면

Cinder = EBS 비슷한 개념

Horizon = AWS Console 비슷한 웹 UI

중요한 실무 포인트

운영하면서 복잡도가 커지는 순서는 보통 이렇습니다.

가장 단순: Horizon, Glance

중간: Keystone

복잡: Nova, Cinder

가장 체감 난이도 높음: Neutron

특히 Neutron은 실제 네트워크 오버레이, OVS/OVN, 라우팅, NAT, DHCP, Floating IP까지 걸리기 때문에 처음 OpenStack를 배울 때 가장 어렵게 느껴지는 경우가 많습니다. Neutron 공식 문서도 여러 프로세스와 플러그인/에이전트 조합으로 동작하는 독립 서비스로 설명합니다.


이제 사용자가 Horizon에서 “VM 생성” 버튼을 눌렀을 때 내부에서 어떤 순서로 흘러가는지, 서비스 호출 흐름으로 설명드리겠습니다.

가장 먼저 한 줄 요약

Horizon은 시작 화면일 뿐이고, 실제 오케스트레이션의 중심은 Nova입니다.
Nova API가 요청을 받고, Keystone으로 인증을 확인하고, Neutron/Cinder/Glance와 연계한 뒤, Scheduler와 Compute가 실제 VM을 띄웁니다. Nova는 API, Scheduler, Conductor, Compute 등의 구성요소로 이루어진 구조이며, 서비스 간에는 DB와 메시지 큐를 사용합니다.

시퀀스 흐름
1) 사용자가 Horizon에서 VM 생성 입력

사용자는 Horizon 웹 콘솔에서 보통 이런 값을 넣습니다.

인스턴스 이름

이미지 선택

Flavor 선택

네트워크 선택

키페어

필요 시 볼륨 부팅 여부

Horizon은 OpenStack 서비스들의 API를 소비하는 웹 UI이며, Nova와 Neutron 같은 서비스 기능을 화면으로 묶어 보여줍니다.

2) Horizon → Keystone 인증 토큰 사용

사용자가 이미 로그인되어 있으면 Horizon은 Keystone에서 발급된 토큰을 사용해 후속 API 요청을 보냅니다.
즉, Horizon이 직접 권한 판단을 하는 것이 아니라, 각 서비스가 Keystone 기반 토큰을 신뢰하는 방식입니다. DevStack 기본 구성에도 Keystone이 포함되어 있고, 다른 OpenStack 서비스의 인증 기반이 됩니다.

3) Horizon → Nova API 로 “서버 생성” 요청

실제 핵심 요청은 Nova Compute API로 갑니다.
즉, 사용자의 “VM 만들기” 액션은 결국 server create 성격의 Nova API 호출입니다. Compute API는 인스턴스 생성과 관리의 공식 API입니다.

4) Nova API 가 요청 검증

Nova API는 다음을 확인합니다.

토큰 유효성

프로젝트/권한

Flavor 사용 가능 여부

요청 파라미터 유효성

이미지/네트워크/볼륨 참조값 확인 준비

Nova API는 HTTP 요청을 받아 명령으로 변환하고, 필요 시 다른 Nova 구성요소와 메시지 큐 또는 HTTP로 통신합니다.

5) Nova → Glance: 이미지 메타데이터 확인

이미지 부팅이면 Nova는 Glance에 선택한 이미지 정보를 조회합니다.

여기서 확인하는 대표 정보:

이미지 존재 여부

포맷

최소 디스크/메모리 조건

아키텍처/속성

Glance는 VM 이미지 관리 서비스이고, Nova 인스턴스 생성 시 이미지 소스로 연계됩니다.

6) Nova → Neutron: 네트워크 포트/네트워크 연결 준비

사용자가 선택한 네트워크에 대해 Nova는 Neutron과 연동해 포트 연결을 준비합니다.

여기서 처리되는 개념:

어느 네트워크에 붙을지

고정 IP 할당

보안그룹 적용

필요 시 Floating IP 연결 준비

DevStack 기본 환경에서도 Neutron과 Floating IP가 포함되며, 인스턴스 네트워크 연결은 Neutron이 담당합니다.

7) 선택적으로 Nova → Cinder: 부팅 볼륨 또는 추가 볼륨 준비

사용자가 “이미지로부터 볼륨 생성 후 그 볼륨으로 부팅”을 선택하면, Nova는 Cinder와 연계합니다.

흐름은 보통 이렇습니다.

Glance 이미지 선택

Cinder가 이미지 기반 부팅 볼륨 생성

Nova가 그 볼륨으로 부팅

공식 사용자 문서에도 이미지에서 부팅 가능한 볼륨을 만들고 그 볼륨으로 인스턴스를 띄우는 절차가 나옵니다.

8) Nova API → DB 기록

Nova는 인스턴스 생성 요청 상태를 DB에 기록합니다.

예:

instance UUID

build 상태

요청한 flavor/image/network 정보

프로젝트/사용자 정보

Nova 아키텍처 문서도 DB를 핵심 구성요소로 설명합니다.

9) Nova API / Conductor → Scheduler: 어느 컴퓨트 노드에 올릴지 결정

그다음 nova-scheduler가 적절한 컴퓨트 호스트를 고릅니다.

판단 요소 예:

CPU/RAM/디스크 여유

가용 영역

호스트 정책

이미지/아키텍처 호환성

Placement 리소스 정보

공식 문서에 따르면 nova-scheduler는 어떤 호스트/노드에 VM을 띄울지 결정합니다.

10) Scheduler 결과 후 → nova-compute 로 작업 전달

대상 호스트가 정해지면 메시지 큐 기반으로 해당 호스트의 nova-compute에 빌드 작업이 전달됩니다.
Nova는 메시지 기반 shared-nothing 구조를 사용하며, compute 서비스가 하이퍼바이저와 실제 통신합니다.

11) nova-compute 가 실제 하이퍼바이저에서 VM 생성

여기가 “진짜 생성” 단계입니다.

nova-compute가 하는 일:

Glance 이미지 가져오기 또는 접근

Neutron 포트 연결

필요 시 Cinder 볼륨 attach

libvirt/KVM 같은 드라이버 호출

디스크/네트워크/메타데이터 구성

가상머신 부팅

Nova는 하이퍼바이저와 VM을 관리하는 컴포넌트를 포함하며, underlying virtualization mechanism과 드라이버로 연동합니다.

12) VM 부팅 완료 후 상태 갱신

성공하면 Nova가 인스턴스 상태를 ACTIVE 같은 상태로 갱신하고, 실패하면 ERROR 등으로 반영합니다.
이 상태는 DB에 반영되고, Horizon이 다시 조회해서 화면에 보여줍니다.

텍스트 시퀀스 다이어그램으로 보면
[사용자]
   ↓
[Horizon]
   ↓  (Keystone 토큰 사용)

   ├─→ [Glance] 이미지 정보 조회

Keystone = 출입증 검사

Nova API = 접수 창구

Glance = OS 템플릿 창고

Neutron = 네트워크 연결 담당

Cinder = 디스크 담당

nova-compute = 실제로 VM을 만들어 올리는 작업자

중요 포인트 3개

첫째, Horizon은 실제 실행 엔진이 아닙니다.
UI일 뿐이고, 진짜 조정자는 Nova입니다.

둘째, Nova 혼자서 VM을 만드는 게 아닙니다.
이미지는 Glance, 네트워크는 Neutron, 볼륨은 Cinder와 협업합니다. DevStack 기본 구성도 이 조합을 전제로 합니다.

셋째, 실제 “스케줄링”과 “생성”은 분리되어 있습니다.
Scheduler가 위치를 정하고, Compute가 생성합니다

이제부터가 OpenStack에서 가장 “클라우드 같다”라고 느껴지는 부분입니다.

Neutron은 단순히 DB에 network/subnet/router 정보를 저장하는 서비스가 아니라, 실제 Linux 네트워크 스택 위에 브리지·포트·라우팅·NAT·DHCP·보안 규칙을 만들어서 VM 통신이 되게 하는 컨트롤 플레인입니다. 공식 문서도 Neutron을 API 서버(neutron-server)와 여러 L2/L3/DHCP/metadata 에이전트로 이루어진 구조로 설명합니다.

큰 구조부터 보면

Neutron에서 보통 등장하는 실행 주체는 이렇습니다.

neutron-server

L2 agent

L3 agent

DHCP agent

metadata agent

ML2 plugin / mechanism driver

백엔드로 OVS 또는 OVN 같은 가상 네트워킹 계층

공식 설치/아키텍처 문서는 neutron-server가 API를 받고, 실제 네트워크 설정 반영은 agent들과 backend가 담당한다고 설명합니다. 또한 message queue가 neutron-server와 various agents 사이에서 사용된다고 안내합니다.

사용자가 네트워크 하나 만들면 내부에서 무슨 일이 생기나

예를 들어 사용자가 Horizon이나 CLI에서 이런 걸 만든다고 가정해보겠습니다.

Network: private-net

Subnet: 10.0.0.0/24

Router: r1

Router gateway: public-net

VM 1대 생성

Floating IP 연결

겉으로는 리소스 몇 개 추가한 것처럼 보이지만, 내부적으로는 Linux 네트워크 구성요소가 꽤 많이 만들어집니다.

1) Neutron Server: “의도”를 받는 API 계층

사용자가 network create, subnet create, router create를 하면 먼저 **neutron-server**가 요청을 받습니다.
여기서 하는 일은 대략 이렇습니다.

Keystone 토큰 검증

DB에 network/subnet/router/port 상태 기록

ML2 plugin 통해 어떤 네트워크 백엔드에 반영할지 결정

필요한 agent에 작업 이벤트 전달

즉 neutron-server는 **“네트워크 컨트롤러”**이고, 실제 리눅스 네트워크 명령을 직접 다 때리는 주체는 아닙니다. 공식 문서도 server는 API/논리 제어, agent는 실제 반영 쪽이라고 설명합니다.

2) L2 계층: Linux bridge / OVS / OVN 으로 포트와 스위치 구성

여기서 중요한 선택지가 나옵니다.
Neutron은 내부적으로 가상 스위치를 구현해야 하는데, 보통 다음 계열이 쓰입니다.

Linux bridge

Open vSwitch (OVS)

OVN(OVS 기반의 더 발전된 SDN 제어 방식)

OpenStack Networking 문서는 ML2와 mechanism driver를 통해 여러 backend를 붙일 수 있고, OVS/OVN 같은 구현과 연계된다고 설명합니다.

Linux bridge 방식이면

각 네트워크를 Linux bridge로 표현

VM vNIC가 tap 디바이스로 bridge에 연결

브리지끼리 VLAN/VXLAN 등으로 연동 가능

OVS 방식이면

br-int 같은 integration bridge

필요 시 br-tun, br-ex 같은 bridge

VM tap 포트를 OVS 포트로 연결

OpenFlow 기반 룰 적용 가능

OVN 방식이면

중앙에서 논리 스위치/논리 라우터를 정의

각 노드의 OVS에 흐름 규칙 자동 배포

전통적인 L3/DHCP agent 의존도를 줄이는 구조

공식 설치 문서와 OVN/OVS 관련 설명은 OVS bridge와 OVN 논리 네트워크 개념을 기준으로 배포 예시를 제공합니다.

3) VM 생성 시 포트(port)는 실제로 “가상 NIC 연결점”

OpenStack에서 port는 매우 중요합니다.
사용자는 그냥 “이 VM을 private-net에 붙인다”고 생각하지만, 내부적으로는 Neutron이 포트 객체를 만들고, 그 포트에 MAC 주소·고정 IP·보안그룹 정보가 붙습니다. API 참조도 port를 핵심 리소스로 정의합니다.

실제 OS 수준에서는 대개 이런 식입니다.

VM 쪽에는 tapXXXX

하이퍼바이저/libvirt 쪽에서 vNIC 연결

tap 장치가 Linux bridge 또는 OVS bridge에 연결

해당 포트에 보안 규칙, VLAN/VXLAN 태깅, anti-spoofing 룰 적용

즉, Neutron port = DB 레코드가 아니라 실제 가상 NIC의 제어 단위입니다.

4) DHCP Agent: VM에 IP를 주는 역할

서브넷을 만들면 보통 VM이 부팅할 때 IP를 자동으로 받습니다.
이 기능은 전통적 Neutron 아키텍처에서 DHCP agent가 담당합니다. OpenStack 문서는 DHCP agent가 dnsmasq 등을 사용해 네트워크별 DHCP 서비스를 제공한다고 설명합니다.

내부적으로는 보통:

네트워크별 DHCP namespace 생성

그 안에서 dnsmasq 실행

VM이 DHCP discover

IP/게이트웨이/DNS 응답

즉 사용자는 그냥 “서브넷 만들었더니 IP가 붙네?”라고 보지만, 내부적으로는 namespace 안에서 DHCP 프로세스가 동작하는 구조입니다.

5) L3 Agent: Router, NAT, Floating IP 의 핵심

Neutron에서 가장 헷갈리는 부분이 router와 Floating IP입니다.

전통적인 Neutron 구조에서 L3 agent는 Linux network namespace와 iptables/nftables 등을 활용해 다음을 처리합니다.

tenant subnet 간 라우팅

external network로의 default gateway

SNAT

DNAT / Floating IP 매핑

OpenStack 보안/아키텍처 문서는 라우팅과 NAT 기능이 L3 계층에서 수행되며, network namespaces를 사용한다고 설명합니다.

router를 만들면 실제로

보통 Linux에 router namespace가 생깁니다.

예:

qrouter-<uuid>

이 namespace 안에 여러 인터페이스가 붙습니다.

내부 subnet 쪽 인터페이스

외부망(public-net) 쪽 인터페이스

그 namespace 안에서는 Linux kernel routing이 실제로 동작합니다.
즉 OpenStack router는 “가상 개념”이 아니라 리눅스 네임스페이스 안의 라우터에 가깝습니다.

6) Floating IP 는 실제로 DNAT/SNAT 매핑

사용자가 VM에 Floating IP를 붙이면, 느낌상 AWS Elastic IP처럼 보입니다.
내부적으로는 보통 L3 계층에서 외부 IP ↔ 내부 fixed IP 매핑이 생깁니다.

예:

Floating IP: 203.0.113.20

Fixed IP: 10.0.0.15

그러면 외부에서 203.0.113.20으로 들어온 트래픽이 router namespace에서 DNAT 되어 10.0.0.15로 전달됩니다. 반대로 나가는 패킷도 적절히 SNAT 처리됩니다. OpenStack Networking 관련 문서와 보안 가이드는 router namespace/NAT 처리 개념을 설명합니다.

즉, Floating IP는 “VM NIC에 공인 IP를 직접 꽂는 것”이라기보다, Neutron 라우터가 NAT로 외부 주소를 대리 노출하는 방식으로 이해하면 가장 정확합니다.

7) Security Group 은 iptables/nftables 레벨의 필터 규칙

사용자는 보안그룹을 “포트 22 허용”, “포트 80 허용”처럼 보지만, 실제로는 VM 포트 단위에 stateful firewall 규칙이 적용됩니다. OpenStack 문서는 security group이 포트에 적용되는 가상 방화벽 개념이라고 설명합니다.

구현 계층은 배포 방식에 따라 다르지만 보통:

iptables 기반 필터

OVS firewall driver

OVN ACL

등으로 반영됩니다. 즉 Security Group = Neutron 정책 → 실제 패킷 필터 룰 변환 과정입니다.

8) Metadata Agent: VM 안에서 169.254.169.254 접근 가능하게 해줌

OpenStack VM 안에서는 보통 메타데이터 서비스 주소로 인스턴스 정보나 user-data를 조회할 수 있습니다. 이건 AWS의 IMDS와 비슷한 개념입니다. 이를 위해 전통 구조에서는 metadata agent가 동작하며, Neutron과 Nova 메타데이터 요청을 중계합니다. 공식 문서가 metadata agent와 metadata proxy 구성을 설명합니다.

즉, VM 안에서 cloud-init이

hostname

SSH key

user-data 스크립트

등을 받는 배경에도 Neutron 측 metadata 경로가 관여합니다.

9) VLAN / VXLAN / Geneve 같은 오버레이

OpenStack가 여러 컴퓨트 노드에 걸쳐 VM 네트워크를 만들 수 있는 이유는, 단순 브리지 하나가 아니라 tenant network를 오버레이로 확장할 수 있기 때문입니다. 설치/아키텍처 문서는 provider network와 self-service network, 그리고 VXLAN/Geneve 같은 overlay network 개념을 설명합니다.

쉽게 말하면:

같은 노드 안에서는 가상 스위치 브리징

다른 노드 간에는 VXLAN/Geneve 터널

각 VM은 같은 L2 네트워크처럼 보임

즉 Neutron은 단순히 IP만 관리하는 게 아니라, 분산 가상 스위칭 fabric까지 다룹니다.
